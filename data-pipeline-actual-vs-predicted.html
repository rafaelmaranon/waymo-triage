🔴 발견 1: subarray().buffer 가 2.16 GB를 낭비하고 있다

핵심 원인: convertRangeImageToPointCloud()에서 new Float32Array(maxPoints × POINT_STRIDE)로 최대 크기를 할당한 후 output.subarray(0, pointCount × POINT_STRIDE)로 유효 부분만 참조하지만, cloud.positions.buffer로 transfer할 때 전체 underlying ArrayBuffer가 전송된다. 빈 픽셀(range=0)이 차지하는 공간이 그대로 메모리에 남는다.

수정 방법 (1줄): rangeImage.ts의 return 직전에 trimmed copy 생성:
const trimmed = new Float32Array(output.buffer, 0, pointCount * POINT_STRIDE) → const trimmed = output.slice(0, pointCount * POINT_STRIDE)
이렇게 하면 2.16 GB → 0 낭비 제거. Steady-state: 3.29 GB → ~1.1 GB 예상.

⚠️ 발견 2: Worker에서 performance.memory가 동작하지 않음

모든 Worker 스냅샷이 0 B를 보고. performance.memory는 Chrome의 main thread 전용 API. Web Worker에서는 undefined를 반환하여 0으로 fallback된다. Worker의 실제 메모리(BROTLI 해제 중 ~100-160 MB/worker)는 측정 불가.

🛠 수정 제안: Float32Array.slice() — 2.16 GB 절약

rangeImage.ts 의 convertRangeImageToPointCloud() 마지막 부분:

현재 (subarray — buffer 공유):
const positions = output.subarray(0, pointCount * POINT_STRIDE)

수정 (slice — 독립 복사본):
const positions = output.slice(0, pointCount * POINT_STRIDE)

효과: 유효 포인트만 포함한 새 Float32Array 생성. 원본 maxPoints 배열은 GC 대상이 됨.
Worker 내부에서 잠시 두 배 메모리(원본 + 복사본)가 존재하지만, transfer 후 원본은 즉시 해제.
예상 절감: 3.29 GB → ~1.1 GB (−2.16 GB, −66%)
Measured with Chrome performance.memory API · --enable-precise-memory-info flag recommended for higher accuracy
